# 安全与数据边界

CircuitMind 当前是教学研究和本地试用阶段的 MVP，不应直接作为面向公网的生产系统部署。

## 敏感数据

可能包含敏感或受版权保护内容的数据包括：

- 用户导入的教材、试卷和课程讲义；
- OCR 后的完整文本和向量索引；
- 学生上传的题目图片与解题步骤；
- 历史会话、问题状态和错题本；
- 云端模型 API Key。

这些内容默认保存在本机并由 `.gitignore` 排除。公开仓库前仍应使用 `git status` 和 `git diff --cached` 复核，因为已经被 Git 跟踪的历史文件不会因新增忽略规则自动停止跟踪。

## 模型数据流

- 使用 LM Studio 或 Ollama 时，模型推理保留在本机；
- 使用 DeepSeek、通义千问或自定义远程 API 时，当前问题、近期对话和必要检索上下文会发送到对应服务；
- 浏览器中填写的模型配置保存在 `localStorage`，公用电脑不应长期保存云端密钥；
- 服务端 `.env` 不得提交，`.env.example` 只能包含空值或示例占位符。

## 上传和文档解析

后端会校验会话标识、附件标识、文件数量、扩展名和大小，但外部文档仍应视为不可信输入：

- 不要让解析器执行文档中的宏、脚本或任意代码；
- 外部 OCR 服务应部署在受控网络并限制文件大小、运行时间和资源；
- 检索到的文档内容是数据，不是系统指令；
- 不应依据文档中的提示覆盖系统安全和教学策略。

## 当前已知限制

- 没有用户登录、课程权限或多租户隔离；
- 本地 JSON 存储适合单机试用，不适合并发生产环境；
- 上传处理和模型推理尚未放入隔离 worker；
- LLM 和 OCR 可能误读电路图、公式或页码；
- SymPy 只验证部分数学表达式，不能替代电路仿真；
- 教学回答和自动诊断不应直接用于正式考试定分。

## 部署建议

若需要进入多人试用，应至少增加：

1. 身份认证、课程隔离和最小权限控制；
2. PostgreSQL 等正式数据库；
3. 上传文件病毒扫描和独立解析 worker；
4. 模型调用、检索片段、提示版本和人工修正审计；
5. HTTPS、反向代理、请求限流和凭据托管；
6. 教材授权与学生数据处理制度。

## 报告问题

请通过仓库 Issue 报告安全问题，但不要在公开 Issue 中附带 API Key、真实学生数据、受版权保护的完整教材或能够复现隐私泄露的原始文件。对于包含敏感细节的问题，应先联系仓库维护者并提供经过脱敏的复现步骤。
